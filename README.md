# 건강검진 AI 병원 추천 시스템

환자가 지역 / 검진유형 / 필수 장비(CT·MRI·PET-CT 등) / 세부 검사항목을 체크하면,
규칙기반 필터링 + 스코어링으로 후보 병원을 추리고, Gemini API로 "AI 추천 이유"와
"병원 간 AI 비교 총평"을 생성해주는 Streamlit 앱입니다.

## 파일 구성

| 파일 | 역할 |
|---|---|
| `korea_regions.py` | 전국 17개 시/도 + 소속 시/군/구 마스터 데이터 (지역 선택 UI 기준) |
| `database.py` | SQLite 스키마 정의, 마스터데이터(장비/검진유형/검사항목), 조회 함수 |
| `sample_data.py` | 구조 검증용 샘플 병원 25곳 데이터 생성 (엑셀 데이터가 없을 때 데모용) |
| `generate_excel_template.py` | 데이터 입력용 엑셀 템플릿(`병원데이터_템플릿.xlsx`) 생성 |
| `excel_to_db.py` | 채워진 엑셀 템플릿을 읽어 SQLite DB로 적재하는 변환 스크립트 |
| `scoring_engine.py` | 하드 필터(지역/검진유형/필수장비/가격대) + 소프트 스코어링(0~100점) 로직 |
| `ai_advisor.py` | Gemini API 연동 - 개별 추천이유 / 병원간 비교 총평 생성 (키 없으면 규칙기반 폴백) |
| `app.py` | Streamlit 메인 UI (체크리스트 입력 → 결과 카드 → 비교 분석) |

## 엑셀로 병원 데이터 입력하기 (신규)

병원 데이터는 엑셀로 관리하고, 앱은 항상 DB(SQLite)만 조회하는 구조입니다.
엑셀은 "입력/관리용", DB는 "실행용"으로 역할을 분리했습니다.

**중요: `app.py`가 실행될 때마다 `병원데이터_템플릿.xlsx` 파일이 코드와 같은
폴더(=git 리포지토리)에 있는지 자동으로 확인합니다.**
- 있으면 → 그 엑셀 내용으로 DB를 새로 구축 (샘플 데이터 대신 실제 데이터로 동작)
- 없으면 → `sample_data.py`의 데모용 샘플 데이터로 동작

즉, **엑셀 파일을 채운 뒤 다른 코드 파일들과 함께 git에 그대로 올리기만 하면**
Streamlit Cloud가 재배포될 때 자동으로 반영됩니다. 터미널 접속이나 별도 명령 실행이
필요 없습니다. (`excel_to_db.py`를 직접 실행하는 것은 로컬에서 미리 결과를
확인해보고 싶을 때만 사용하면 됩니다)

### 로컬 작업 순서

```bash
# 1) 빈 템플릿 생성 (최초 1회, 예시 1행 포함)
python generate_excel_template.py
# → 병원데이터_템플릿.xlsx 생성됨

# 2) 엑셀 파일을 열어 노란색 예시 행 아래부터 실제 데이터 입력
#    (시트: 1_병원마스터 / 2_보유장비 / 3_검진유형가격 / 4_검사항목)
#    - 시도/장비종류/검진유형/검사항목명은 드롭다운 목록에서만 선택
#    - 병원ID는 4개 시트에서 동일하게 사용해 서로 연결

# 3) (선택) 로컬에서 미리 결과를 확인하고 싶다면 직접 변환해보기
python excel_to_db.py 병원데이터_템플릿.xlsx
streamlit run app.py   # 로컬 브라우저에서 확인

# 4) 문제 없으면 병원데이터_템플릿.xlsx 파일을 다른 코드 파일들과 함께 그대로 git에 업로드
#    (health_checkup.db 파일은 올리지 않습니다 - 앱이 실행될 때 엑셀로부터 자동 재생성됩니다)
```

### 주의사항

- **예시 행만 있는 빈 템플릿을 실수로 그대로 올리지 마세요.** 노란색 예시 1행만
  들어있는 상태로 올리면, 앱이 "엑셀 데이터가 있다"고 판단해서 샘플 25곳 대신
  예시 병원 1곳만 보여주게 됩니다.
- git 리포지토리를 **비공개(Private)로 설정하는 것을 권장**합니다. 병원별
  보유 장비·가격·검사항목 커버리지 데이터는 조사에 시간을 들여 만든 서비스의
  핵심 자산이므로, 공개 리포지토리에 그대로 노출할 필요는 없습니다.
- `excel_to_db.py`는 실행(=앱 재시작) 때마다 기존 DB를 지우고 엑셀 내용으로
  새로 채웁니다. 실행 후 아래처럼 처리 결과와 경고를 알려줍니다.

```
✅ 병원 2곳 / 장비 3건 / 검진유형·가격 3건 / 검사항목 매핑 3건 적재 완료
⚠️  경고 1건 (아래 항목은 건너뛰었습니다):
  - [검사항목] 병원ID 2: 검사항목명 '존재하지않는검사항목'이 마스터 목록에 없어 건너뜁니다.
```

앱 화면 상단에도 이 경고가 자동으로 표시되므로, 배포 후에도 데이터에 문제가
있는지 바로 확인할 수 있습니다.



## 실행 방법

```bash
pip install -r requirements.txt
streamlit run app.py
```

### Gemini API 키 설정 (선택)
키가 없어도 앱은 정상 동작하며, AI 설명 부분만 규칙기반 요약으로 대체됩니다.

`.streamlit/secrets.toml` 파일 생성:
```toml
GEMINI_API_KEY = "your-api-key-here"
```
또는 환경변수 `GEMINI_API_KEY`로도 설정 가능합니다.

## 추천 로직 설계

**하드 필터 (반드시 충족)**
- 지역(시/도, 선택 시 구/군까지)
- 선택한 검진유형을 실제로 제공하는지
- 체크한 필수 장비를 **모두** 보유하는지

**소프트 스코어링 (100점 만점, 가중치는 `scoring_engine.py` 상단 상수에서 조정 가능)**
| 항목 | 배점 |
|---|---|
| 지역 근접성 (구/군 정확 일치) | 20점 |
| 요청 검사항목 커버리지 | 40점 |
| 필수장비 외 추가 보유 장비 다양성 | 15점 |
| 인증/신뢰도 (JCI, 우수기관 등) | 10점 |
| 후보군 내 상대적 가격 경쟁력 | 15점 |

## 실서비스 전환 시 체크리스트

1. **병원 마스터 데이터**: 국민건강보험공단 공공데이터포털의 "건강검진기관 지정 현황"
   API로 병원명/주소/전화번호/검진유형 기본값을 채우고, 조정윤님이 그동안 다른
   프로젝트에서 다뤄오신 심평원/식약처 API 연동 패턴을 그대로 재사용할 수 있습니다.
2. **장비/검사항목 데이터**: 공공데이터에는 보유 장비 스펙이 나오지 않으므로, 병원
   홈페이지 조사 또는 병원 제휴를 통해 실측 데이터로 `hospital_equipment`,
   `hospital_exam_items` 테이블을 채워야 합니다. (이 부분이 서비스의 핵심 자산이 됩니다)
3. **DB 마이그레이션**: 스키마를 표준 SQL로 작성했기 때문에 Supabase(PostgreSQL)로
   옮길 때 `CREATE TABLE` 구문을 그대로 사용하고, `database.py`의 커넥션 함수만
   `psycopg2`/`supabase-py` 클라이언트로 교체하면 됩니다.
4. **가격 최신화**: 가격은 변동이 잦으므로, 관리자용 별도 Streamlit 페이지(또는 탭)로
   병원 담당자가 직접 가격을 업데이트할 수 있는 CRUD 화면을 추가하는 것을 권장합니다.
5. **AI 비용 관리**: 현재 구조는 "버튼 클릭 시에만" Gemini를 호출하도록 지연 생성
   방식으로 설계되어 있어, 검색 결과 노출만으로는 API 비용이 발생하지 않습니다.

## 향후 확장 아이디어 (차별화 포인트)

- **연령/성별/가족력 기반 맞춤 검진 추천**: 국가암검진 권고안 등을 반영해 "이 나이대는
  이 검사가 꼭 필요합니다" 같은 사전 가이드 제공
- **장비 최신성 신뢰도 표시**: `install_year`가 오래된 장비는 "도입 10년 이상" 뱃지로
  투명하게 표기
- **결과 상담 방식 비교**: 검진 후 전문의 대면 상담 여부, 결과지 수령 방식(온라인/우편)
  등 검진 "경험" 측면의 차별점 추가
- **즐겨찾기/저장**: 로그인 연동 시 사용자별 관심 병원 저장 및 재방문 시 불러오기
